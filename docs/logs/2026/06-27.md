## 17:37 - Refactor BlockReader trait to GAT (Phase 1/3/4 of unify-block-io plan)

**What**: Refactored `BlockReader` trait to use a Generic Associated Type (`Guard<'a>`) so the read path can serve both owned bytes (LSM buffered I/O) and future zero-copy pinned reads (WAL O_DIRECT cache) through one abstraction.
**Why**: The original `BlockReader` returned a fixed `Bytes`, which cannot represent WAL's borrowed `PinGuard`. Unifying via GAT lets upper layers depend only on `Deref<Target=[u8]>`. This is the prerequisite for the WAL index read-path work.
**How**:
- `block.rs`: `BlockReader` now declares `type Guard<'a>: Deref<Target=[u8]> + AsRef<[u8]> where Self: 'a`; `FileBlockReader`/`InMemoryBlockReader` set `Guard = Bytes`; `Arc<R>` impl forwards the associated type.
- `iterators/storage_iter.rs`: `IndexBlockIter::from_block` / `DataBlockIter::from_block` made generic over `B: AsRef<[u8]>` (was `Bytes`).
- `iterators/block_iter.rs`: `NormalBlockIter::new` takes `impl AsRef<[u8]>` and copies to owned `Bytes` internally — it already stored owned bytes, so no iterator restructuring was needed.
- `iterators/data_entry_decode_iter.rs`: `from_block` updated to the generic signature.
- Added `test_arc_block_reader` to verify `Arc<R>` works under GAT.

**Actual approach vs plan**: The plan proposed restructuring iterators to "cache parsed entries". The simpler real fix works because `NormalBlockIter` already owned its block — a guard is consumed (copied into `Bytes`) at the `from_block` call site then dropped, so `PinGuard` unpinning happens automatically and the WAL deadlock discipline holds by construction.

**Deferred**: Phase 2 (WAL `SegmentBlockReader` adapter) — user instructed not to change WAL yet. Will be done when WAL O_DIRECT is unified with LSM in a follow-up.

**Verification**: ✅ `cargo build --workspace` passes; ✅ `cargo test --workspace` all pass (155 storage lib tests + all integration tests); ✅ new `test_arc_block_reader` passes; ✅ `cargo clippy -p storage` shows no new warnings (only pre-existing dead-code warnings in disabled modules).

**Files changed**: block.rs, iterators/storage_iter.rs, iterators/block_iter.rs, iterators/data_entry_decode_iter.rs, lsm_iter.rs (unused imports).

## 18:10 - Make NormalBlockIter generic over storage type (zero-copy, approach A)

**What**: Changed `NormalBlockIter` from a fixed `Bytes` holder to `NormalBlockIter<B: Deref<Target=[u8]> = Bytes>`, so the block-format iterator stores its block source directly — no per-block copy.
**Why**: The previous GAT step used `from_block(impl AsRef<[u8]>)` + `Bytes::copy_from_slice`, which introduced a 4 KB memcpy + alloc on every block load, even for the LSM path where `FileBlockReader` already returns an owned `Bytes`. The user flagged this as a real hot-path regression. Settled on approach A (hold the source generically) over a stopgap `Into<Bytes>` (which would have copied WAL's `PinGuard` and abandoned the zero-copy goal).
**How**:
- `block.rs`: `BlockReader::Guard<'a>: Deref<Target=[u8]>` only (dropped `AsRef`).
- `storage_iter.rs`: `IndexBlockIter`/`DataBlockIter` gain `type Block: Deref<Target=[u8]>`; `from_block(block: Self::Block)`.
- `block_iter.rs`: `NormalBlockIter<B>` stores `block: B`; `new(block: B)` no longer copies. `parse_header` uses `self.block.deref()`.
- `index_tree_iter.rs` / `sst_iter.rs`: bound `for<'a> R::Guard<'a>: Into<I::Block>` (and `Into<D::Block>`), `.into()` at the `from_block` call sites. For LSM, `Bytes: Into<Bytes>` is the blanket identity move → **zero copy**.
- Added `test_generic_storage_vec`: iterates a `Vec<u8>`-backed block, proving `NormalBlockIter<B>` is not pinned to `Bytes` (de-risks the future WAL `PinGuard` path).

**Borrowed-guard caveat**: holding a real `PinGuard<'r>` is still blocked by self-reference (`read_block(&self) -> Guard<'_>` ties the guard to `&self`); that lifetime-on-struct redesign is deferred to WAL Phase 2.

## 19:05 - WAL PinGuard: extract DiskManagerInner, make guard 'static (Q1)

**What**: Refactored WAL `DiskManager` to hold `Arc<DiskManagerInner>`; `PinGuard` now holds `Arc<DiskManagerInner>` instead of `&'a DiskManager`, making it `'static`.
**Why**: A borrowed `PinGuard<'a>` (for Drop's unpin) made a stateful iterator that stores it self-referential — blocking zero-copy cached reads. `'static` (via `Arc`) dissolves that: the guard can be stored in `NormalBlockIter<PinGuard>` with no self-reference, no materialization, no iterator-stack rewrite. This is the safe version of the raw-pointer fix: refcount-enforced liveness instead of an unchecked engine invariant + `unsafe` Drop.
**How**:
- Extracted `DiskManagerInner { block_size, cache, cv }`; `DiskManager { inner: Arc<DiskManagerInner> }`.
- `read_block` returns `'static` `PinGuard` (clones the Arc); body uses `self.inner.*`.
- `PinGuard { inner: Arc<DiskManagerInner>, idx, ptr, len }` — no lifetime param. `Drop` is now fully safe (no `unsafe`); the only remaining `unsafe` is the existing buffer `Deref` (raw ptr into the fixed `Box<[Frame]>`, justified by Arc-alive + pin).
- Fixed stale doc comments (`PinGuard<'a>` → `PinGuard`).
- Added compile-time proof test `pin_guard_is_static_and_derefs`: asserts `PinGuard: 'static`, `PinGuard: Deref<Target=[u8]>`, and `NormalBlockIter<PinGuard>` is well-formed.

**Not done (deliberate)**: did NOT add per-frame `RWMutex` (Q2) — the cached blocks are immutable after load, so the read/write latch layer is unused; `pin_count` (= buffer-fix) is the right and sufficient protection. A latch layer earns its keep only when the cache serves mutable pages (future write cache).

## 21:42 - WalIndexReader (Phase 1.2) + fix two latent SST deep-tree bugs

**What**: Implemented `WalIndexReader` — the `BlockReader` adapter over the WAL `DiskManager` that serves an index SST's blocks zero-copy through the CLOCK cache as a `'static` `PinGuard`. While verifying it end-to-end (`SstIter<WalIndexReader, NormalBlockIter<PinGuard>, …>` round-trip), uncovered and fixed two independent latent correctness bugs in the existing SST stack that only surface on deep index trees (height ≥ 4):
1. `SstBuilder::finish` left the topmost index level unflushed when it held >1 entry → `root_position` reached only the last top-level block, orphaning the rest → `seek_to_first` landed near the END of the key space.
2. `IndexTreeIter::inner_next`, on a step that exhausted the leaf AND its parent in one call, re-descended by calling `next()` on a freshly `seek_to_first`'d child → skipped that child's first entry (and its whole subtree).

**Why**: `WalIndexReader` is the Phase 1.2 primitive of the WAL index read path — it proves the GAT/PinGuard pipeline composes end-to-end. The round-trip test used `block_size=64` (→ height 6), which was the first time anyone built a tree tall enough to trigger the two bugs. Existing SST tests max out at height ≤ 3 (single-level index pops only), so the bugs passed unnoticed. Both affect ANY deep SST tree — including large LSM SSTs from compaction, not just the WAL index — so they were serious latent corruption, not a WAL-specific issue.

**How**:
- `wal/index_reader.rs` (new): `WalIndexReader { dm: Arc<DiskManager>, file_id, fd }`, `type Guard<'a> = PinGuard`, `read_block` maps a block-aligned `Position.offset → block_idx`. Borrows the fd (caller owns the segment's `idx_fd`); footer is a separate one-shot buffered read at open time.
- `errors.rs`: `From<WalError> for StorageError` so WAL primitives can back `BlockReader` (which returns `StorageResult`).
- `builder/sst.rs`: after the bottom-up flush loop, flush the topmost builder if `entry_count() > 1` so a single root block is always produced.
- `iterators/index_tree_iter.rs`: rewrote `inner_next` as two phases — (1) advance current level, walking up while exhausted; (2) re-descend to the leaf through first-entries (mirroring the correct `inner_seek_to_first`), never calling `next()` on a just-pushed level.

**Not done (deliberate)**: ~~reverse `inner_prev`~~ — originally noted as a forward-semantics copy-paste left untouched. On closer inspection that characterization was wrong: reverse *does* work on shallow trees (`.next()`/`.seek_to_first()` resolve to `ReverseIter` in the reverse impl block); it just had the **same** multi-level-pop skip bug as `inner_next`. Fixed it for correctness/consistency (LSM reverse range scans over large SSTs would hit it too), with regression test `test_reverse_deep_tree_full_scan_reaches_all_keys`. See bug note addendum.

**Verification**: ✅ `cargo test -p storage --lib` — 162 passed, 0 failed (incl. new `sst_roundtrip_through_cache`, `read_block_returns_cached_pin_guard`, `test_deep_tree_full_scan_reaches_all_keys`, and `test_reverse_deep_tree_full_scan_reaches_all_keys` regressions). ✅ `cargo build --workspace` clean (5 pre-existing warnings, none new).
