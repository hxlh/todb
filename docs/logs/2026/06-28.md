# 2026-06-28 Dev Log

## 16:42 - WAL index `.idx` SST now sealed via O_DIRECT (`ODirectBlockWriter` + `ODirectSstWriter`)

**What**: Replaced the buffered `block::FileBlockWriter` in the WAL index seal path with a WAL-internal O_DIRECT writer pair that writes the `.idx` SST via `O_DIRECT` (no page cache). Reverses the Phase 2 sub-plan decision "write side stays buffered" — the user asked the WAL to own a direct-I/O block writer rather than reach into the LSM/block layer's `FileBlockWriter`.
- `wal/odirect_writer.rs` (new): **two types**, split to honor the `SstWriter` contract (sst_writer.rs: "builder encodes; the raw writer only asserts the block_size invariant and writes"):
  - `ODirectBlockWriter` (dumb `BlockWriter`): opens an `O_DIRECT` fd (`open_fd(path, o_direct)`), asserts input is `block_size`, copies into a `block_size`-aligned `AlignedMem`, `pwrite`s at the next block-aligned offset. **No SST/footer knowledge** — just writes bytes (same role as `FileBlockWriter`/`InMemoryBlockWriter`).
  - `ODirectSstWriter` (`SstWriter`, wraps `ODirectBlockWriter`): data/index blocks delegate verbatim; `write_footer` does the one piece of O_DIRECT encoding the raw writer must not — see below.
- **Variable-footer vs O_DIRECT alignment** (lives in `ODirectSstWriter::write_footer`, NOT the raw writer): the SST footer (`body ++ [body_len:u32]`) is sub-`block_size`, so it can't go raw through O_DIRECT. `write_footer` packs it into one block as `[body][padding][trailer]` — the `body_len` trailer at the block's last 4 bytes, body at the block start — so a block-aligned read of the final block recovers it (`trailer = block[block_size-4..]`, `body = block[..body_len]`). Single-fd — no O_DIRECT/buffered coherency hazard. (First cut wrongly put this length-branch into `ODirectBlockWriter::write_block`; corrected after review — a raw `BlockWriter` must stay encoding-free, else it breaks the `BlockWriter`/`SstWriter` seam the LSM also relies on.)
- `wal/facade.rs::finalize_segment`: `FileBlockWriter` → `ODirectBlockWriter` + `ODirectSstWriter` (`SstBuilder::new(ODirectSstWriter::new(writer, &option), option)`); imports swapped. `sst_writer.into_inner().sync_all()` still holds (`ODirectBlockWriter::sync_all` = `fdatasync_fd`, preserving the commit invariant: durable `.idx` before `.meta`).
- `wal/segment.rs`: `open_fd` is now `pub(crate)` so the new writer can reuse it.
- `wal/mod.rs`: added `pub mod odirect_writer` + exports `ODirectBlockWriter`, `ODirectSstWriter`.
- `block.rs`: removed `FileBlockWriter::sync_all` (its only caller was the WAL seal path; now dead).
- `wal/index_reader.rs` (test scaffolding) + `tests/wal_index_write.rs::read_footer`: switched to `ODirectSstWriter`/`ODirectBlockWriter` / updated to parse the `[body][padding][trailer]` layout (read the final block, not the file tail). WAL is now `block::FileBlockWriter`-free.

**Why**: The user wanted the WAL index to own a direct-I/O block writer instead of depending on the LSM/block layer's buffered `FileBlockWriter` — keeps WAL write I/O off the page cache and removes a cross-module dependency. The footer layout is the load-bearing detail: O_DIRECT demands aligned writes, so the variable footer must be padded, and the trailer must land at a known offset (block end) for the read path to find it. Split into a dumb `BlockWriter` + an encoding `SstWriter` so the raw writer stays a pure "write bytes" layer — matching the existing `FileBlockWriter`/`InMemoryBlockWriter` role and the documented `SstWriter` contract.

**Verification**: ✅ `cargo build --workspace` clean. ✅ `cargo test -p storage --lib` — 165 passed, 0 failed. ✅ Both end-to-end Phase 2 tests (`flush_seals_idx_sst_and_meta_single_segment`, `flush_seals_one_idx_per_segment_on_rollover`) green with the SST now built via `ODirectSstWriter`/`ODirectBlockWriter`. ✅ Real kernel `O_DIRECT` confirmed on this ext4 env: python O_DIRECT pwrite/pread smoke + `odirect_true_round_trips_on_ext4` (gated on `TODB_ODIRECT_DIR`) round-trips a footer through `ODirectSstWriter::write_footer` with `o_direct=true`. (`o_direct=false` remains the tmpfs/CI mode via `WalConfig.o_direct`.)

**Notes / gotchas**:
- Phase 3 (`SegmentIndex` open) must decode the footer with the **same** `[body][padding][trailer]` layout (read the final `block_size` of the `.idx`), NOT the LSM-style "last 4 bytes of file → body immediately precedes trailer" — that LSM read breaks here because padding sits between body and trailer.
- `o_direct=false` (tmpfs) still uses the padded layout, so the read-side decode is identical regardless of the flag.

Plan: docs/plans/2026-06-27-wal-index-read-path.md (write side now O_DIRECT). Next: Phase 3 — `SegmentIndex` open (`.meta` + `.idx` footer decode in the padded layout) + `get`/`scan`, with a per-segment unique `file_id`/index `DiskManager`.

## 15:03 - WAL index write path (Phase 2): MemTable → `.idx` SST + `.meta`

**What**: Retargeted the WAL flush write path so each segment seals a real `.idx` SST (LSN→(offset,len)) readable by `WalIndexReader` (Phase 1.2), and moved the header to a separate `.meta` file. Deleted the old flat `.idx` format.
- `wal/facade.rs`: `FlushState` now holds `index_mem: MemTable<Bytes,Bytes>` (one per active segment) instead of `IdxTail`+`idx_blocks`. The `flush_buffer` capture loop (`facade.rs:380`) does `index_mem.put(lsn_to_key(lsn), encode_offset_len(base+pos, flen))`. `finalize_segment` seals the MemTable → `.idx` SST via `SstBuilder`+`FileBlockWriter`, `fdatasync`s the SST, THEN writes `.meta` (commit invariant). `ensure_segment` resets the MemTable + writes the create-time `.meta` (`entry_count==0` = active marker). Removed `write_idx_block`.
- `wal/segment.rs`: `idx_fd`→`meta_fd`; `write_idx_header_double`/`read_idx_header`→`write_meta_header_double`/`read_meta_header` (target `.meta`). Segment opens `.log`+`.meta` only (the `.idx` SST is sealed by the flush thread, read via `WalIndexReader` later). Renamed files `wal-{seg_id}.{log,idx}` → `seg_{seg_id:05}.{log,meta,idx}` (zero-padded so a dir listing is in seg_id order — for recovery).
- `wal/index.rs`: added encoding helpers (`lsn_to_key`/`key_to_lsn` BE, `encode_offset_len`/`decode_offset_len` LE) + path helpers (`log_path`/`meta_path`/`idx_path`). Deleted `IdxTail`, `IdxEntry`, `ENTRIES_PER_BLOCK`, `BLOCK_SIZE` + their tests.
- `wal/mod.rs`: updated exports.
- `block.rs`: `FileBlockWriter::sync_all` (fdatasync the `.idx` before `.meta`).
- `memtable.rs`: added `MemTable::len`/`is_empty` (needed for `entry_count`).

**Why**: Phase 1.2 (`WalIndexReader`) had nothing to read — the old `.idx` was a flat array of 20 B `IdxEntry` blocks, unreadable by `SstIter`. Phase 2 makes the flush path produce the SST format so the zero-copy cached read pipeline has real artifacts, and deletes the dead flat-format code.

**How** (key decisions):
- Write side stays **buffered** (`FileBlockWriter`); the SST is sealed once at close, not a hot path. Reads stay O_DIRECT+cache (Phase 3).
- `.idx` is a pure SST from offset 0 (header moved to `.meta`), so `WalIndexReader`'s `offset/block_size` map is exact.
- **Commit ordering**: seal SST → `sync_all(.idx)` → write `.meta` double-write. `entry_count==0` in `.meta` marks a segment active/unsealed — the discriminator recovery (Phase 5) will use.
- Key = LSN 8-byte big-endian; value = `(offset:u64, len:u32)` little-endian (12 B).

New tests in `src/tests/wal_index_write.rs`: (1) append 50 → sync → close → `.meta` header correct + `.idx` SST readable via `WalIndexReader`/`SstIter`, every `(lsn,offset,len)` cross-checked against the `.log` frames; (2) multi-segment rollover → one `.idx`/`.meta` per segment, all 1000 lsns indexed. ✅ `cargo build --workspace` clean.
